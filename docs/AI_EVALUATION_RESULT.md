## 3. 黄金业务场景

### 场景1：流量下降导致销售下降

输入背景：销售额和访客同时下降，支付转化率基本稳定，系统已有销售下降与访客下降 Finding。

预期AI行为：

- 综合两个已有 Finding，而不是逐条改写规则卡片。
- 将流量下降表述为销售承压的可能关联因素。
- 优先组织流量渠道核查、有效流量恢复和销售漏斗复盘。
- 提醒用户当前数据不能证明唯一因果。

不得出现：创建转化异常、编造渠道或流量数字、断言流量下降是唯一原因。

实际结果记录重点：是否引用两个 Finding；是否生成跨问题 Insight；是否保持“可能”语气；行动是否先验证流量再调整投入。

### 场景2：流量稳定但转化下降

输入背景：访客稳定，支付转化率下降，可能伴随销售承压，但没有访客下降 Finding。

预期AI行为：

- 将重点放在流量承接、商品页面、价格权益或购买流程。
- 只从转化相关候选 Cause 和 Action 中选择内容。
- 建议先检查转化漏斗，再决定是否增加流量投入。

不得出现：创建访客下降 Finding、断言商品页面一定有问题、无条件增加广告预算。

实际结果记录重点：是否错误归因流量；验证方法是否覆盖点击、加购和下单环节；行动顺序是否合理。

### 场景3：广告投入增加但销售未增长

输入背景：广告投入增加，销售没有增长，可能同时触发 ROAS 下降。

预期AI行为：

- 综合投入变化与产出效率问题。
- 将流量质量、投放组合和预算分配保持为待验证假设。
- 先安排投放效率复盘，再根据验证结果调整预算。

不得出现：编造广告平台、受众或素材数据；立即停止全部投放；把相关性描述为确定因果。

实际结果记录重点：是否同时引用广告投入与 ROAS Finding；是否区分“增加投入”和“有效增长”；是否提供验证路径。

### 场景4：销售增长但毛利下降

输入背景：销售增长，但毛利率下降并触发规则，核心问题是增长质量而不是销售异常。

预期AI行为：

- 说明销售增长与利润承压可以同时发生。
- 将成本、折扣和商品结构作为候选假设。
- 建议复盘毛利来源，不创建新的销售增长异常。

不得出现：断言折扣是唯一原因；编造成本或折扣数字；建议停止全部促销。

实际结果记录重点：是否准确解释“规模增长但质量下降”；是否保持 Finding 边界；行动是否围绕毛利驱动因素。

### 场景5：重点商品下降并伴随库存风险

输入背景：重点商品销售下降，同一商品库存可售天数偏高，系统已有商品销售与高库存 Finding。

预期AI行为：

- 关联需求走弱与库存消化压力，但不宣称确定因果。
- 优先安排重点商品复盘，然后控制补货或制定库存消化计划。
- 强调重点商品的业务影响和后续观察指标。

不得出现：编造竞品、价格或消费者反馈；直接建议大幅降价清仓；忽略重点商品属性。

实际结果记录重点：两个 Finding 是否正确关联；Cause/Action 是否属于对应商品 Finding；补货和库存行动顺序是否清晰。

## 4. 实际AI报告记录方式

每次更换 Provider、模型、Prompt 或 Schema 后，都要为5个场景分别新增一条评估记录。不得覆盖历史记录。

### 4.1 运行信息

```text
评估日期：
评估人：
Git commit：
场景名称：
Provider：
模型：
Schema版本：
Prompt版本：
生成状态：success / unavailable / provider_error / validation_error
响应耗时：
```

记录中不得包含 API Key、原始 CSV、用户敏感数据或 Streamlit secrets。

### 4.2 自动门禁结果

| 检查项 | 结果 | 备注 |
|---|---|---|
| JSON结构正确 | 通过/失败 | 记录错误字段 |
| Finding引用正确 | 通过/失败 | 记录无效ID |
| Cause引用属于对应Finding | 通过/失败 | 记录无效ID |
| Action引用属于对应Finding | 通过/失败 | 记录无效ID |
| 自然语言未产生业务数字 | 通过/失败 | 记录原句，不记录Key |
| 原因保持假设表达 | 通过/失败 | 记录缺少假设语气的原句 |

自动门禁失败的报告不得进入页面截图或人工质量评分，但需要记录失败原因和降级状态。

### 4.3 实际报告摘要

只记录经过 Validator 的内容：

```text
经营摘要：
覆盖的Finding：
跨问题Insight：
原因假设与验证方式：
行动顺序：
数据限制：
```

完整 JSON 如需留档，应使用不含原始数据和密钥的脱敏文件；默认不提交真实调用日志。

## 5. 人工评分标准

每个维度使用1–5分：

- 1分：严重不符合，可能误导运营人员。
- 2分：多数内容不可用，需要重写。
- 3分：基本可用，但有明显遗漏或空泛内容。
- 4分：符合预期，只有轻微表达问题。
- 5分：完整、可信、具体，可直接用于面试演示。

### 5.1 报告质量评分

| 维度 | 分数 | 评估问题 |
|---|---:|---|
| 事实一致性 | /5 | 是否忠于Finding、Evidence和规则Priority |
| Finding覆盖 | /5 | 是否覆盖重点问题且没有创建新问题 |
| 多问题综合 | /5 | 是否提供规则卡片之外的合理关联 |
| 假设表达 | /5 | 原因是否使用可能、待验证等语气 |
| 验证可执行性 | /5 | 是否明确应该检查什么 |
| 行动具体性 | /5 | 是否选择有效Action并给出清晰顺序 |
| 输入边界遵守 | /5 | 是否没有编造数字、渠道、商品或业务事实 |

通过标准：平均分不低于4分；事实一致性和输入边界遵守不得低于4分；出现新业务数字或新 Finding 时该场景直接不通过。

### 5.2 AI增量价值评分

使用同一场景分别完成“只看规则卡片”和“规则卡片 + AI报告”任务。

| 维度 | 分数 | 记录方式 |
|---|---:|---|
| 是否更快理解问题 | /5 | 记录两种方式的理解用时 |
| 是否更容易决定下一步 | /5 | 记录行动排序是否减少犹豫 |
| 是否降低人工分析成本 | /5 | 记录减少的手工汇总步骤 |
| 是否提供额外价值 | /5 | 判断是否包含关联、假设、验证和行动组织 |

通过标准：平均分不低于4分；“决定下一步”和“额外价值”不得低于4分；AI增强方式的总用时应低于规则卡片方式。

## 6. 自动化评估门禁

对应测试：`tests/test_ai_evaluation.py`。

测试对5个固定场景执行：

```text
Golden Scenario Findings
→ Payload Builder
→ Fake Provider
→ Report Validator
→ Evaluation Gates
```

当前自动检查：

- 顶层 JSON 字段与冻结 Schema 一致。
- 所有 finding_id 都存在于当前输入。
- cause_id 属于对应 Finding 的候选原因。
- action_id 属于对应 Finding 的候选行动。
- AIReport 自然语言字段不包含数字；`action_sequence` 不属于自然语言检查范围。
- 每条原因假设包含“可能”“假设”“待验证”或同类验证语气。

边界说明：自动数字检查不能判断所有输入外事实，自然语言事实一致性和因果表达仍需人工评分。该测试不是生产 Validator，不改变页面展示规则。

运行方式：

```powershell
.\.venv\Scripts\python.exe -m pytest tests\test_ai_evaluation.py -q
```

## 8. 规则诊断 vs AI增强

| 用户问题 | 确定性规则诊断 | AI增强报告 |
|---|---|---|
| 发生了什么 | 用固定公式和阈值生成Finding | 不重新判断，只总结已有Finding |
| 证据是什么 | 展示本期、基准和变化 | 通过finding_id引用，不生成业务数字 |
| 哪个问题优先 | 规则引擎计算Priority | 解释现有Priority，不修改它 |
| 问题是否相关 | 每条规则独立输出 | 综合多个Finding并保持可能关系表达 |
| 为什么发生 | 提供受控Cause候选 | 选择原因假设并给出验证方法 |
| 下一步做什么 | 提供受控Action候选 | 组织Action Sequence并解释顺序 |
| 数据有什么限制 | 规则只说明触发条件 | 明确不能确认的内容和数据边界 |

面试表达重点：TrendPilot 不是让大模型直接读取电商数据，而是让确定性系统负责事实和约束，让 AI 负责综合、解释和行动组织，最后再用 Validator 与人工评估控制输出质量。
